**1. finalize_dispute circuit (encrypted-ixs/src/lib.rs):**

Add after add_dispute_vote, mirrors compute_payouts pattern:

```rust
/// Reveal encrypted dispute vote totals at finalization.
///
/// Takes the encrypted VoteTotals and reveals both weighted vote values as plaintext.
/// The program uses these to determine the winning outcome.
#[instruction]
pub fn finalize_dispute(
    tally_ctxt: Enc<Mxe, VoteTotals>,
) -> (u64, u64) {
    let tally = tally_ctxt.to_arcis();
    (tally[0].reveal(), tally[1].reveal())
}
```

**3. finalize_dispute.rs (queue instruction):**
Mirror compute_payouts.rs exactly:

- Validates `dispute.status == 0` (Voting) and `dispute.vote_count >= dispute.quorum`.
- Checks MPC lock with 60s timeout recovery.
- Sets MPC lock on dispute.
- Transitions dispute status to 1 (Finalizing).
- ArgBuilder: `.account(dispute_tally.key(), 16, 64)` only (no user input).
- Callback accounts: Market (writable), Dispute (writable).

Accounts struct: `#[queue_computation_accounts("finalize_dispute", payer)]`
- `payer: Signer` (mut) -- anyone can trigger finalization once quorum reached
- `dispute: Account<Dispute>` (mut)
- `dispute_tally: Account<DisputeTally>`
- `market: Account<Market>` (validates state == Disputed)
- Standard Arcium accounts
- `system_program`, `arcium_program`

**4. finalize_dispute_callback.rs (handler in lib.rs):**

Callback accounts struct uses `#[callback_accounts("finalize_dispute")]` macro.
Custom callback accounts: Market (mut), Dispute (mut).

On success (after verify_output):
- Output is `FinalizeDisputeOutput { field_0: { field_0: u64, field_1: u64 } }` -- same pattern as compute_payouts.
- `revealed_yes_votes = result.field_0` (stake-weighted yes votes)
- `revealed_no_votes = result.field_1` (stake-weighted no votes)
- If `revealed_yes_votes > revealed_no_votes`: `market.winning_outcome = 1` (Yes wins)
- If `revealed_no_votes > revealed_yes_votes`: `market.winning_outcome = 2` (No wins)
- If tied: DO NOT resolve. Set `dispute.status = 0` (back to Voting). The add_tiebreaker instruction must be called next.
- On non-tie resolution: Set `market.state = 2` (Resolved), `dispute.status = 2` (Settled).
- Clear `dispute.mpc_lock`.

On failure:
- Clear `dispute.mpc_lock`, revert `dispute.status` back to 0 (Voting).

**1. add_tiebreaker.rs:**

Called when finalize_dispute detects a tie (dispute.status was reset to Voting by callback). This instruction:

- Validates `dispute.status == 0` (Voting -- returned to voting after tie).
- Validates `dispute.tiebreaker_added == false` (only one tiebreaker per dispute).
- Selects 1 additional juror from the ResolverRegistry who is NOT already in `dispute.jurors`.
  - Use deterministic selection: `seed = market_id ^ clock.slot ^ dispute.vote_count as u64`.
  - Iterate registry, skip existing jurors, pick first match.
- Pushes new juror pubkey and stake snapshot to `dispute.jurors` and `dispute.juror_stakes`.
- Extends voting window: `dispute.voting_end = clock.unix_timestamp + 86400` (24h extension).
- Increments `dispute.quorum` by 1 (now 6 of 8 required).
- Sets `dispute.tiebreaker_added = true`.
- Increments new resolver's `active_disputes`.

Accounts:
- `payer: Signer` (mut) -- anyone can trigger
- `dispute: Account<Dispute>` (mut)
- `market: Account<Market>` (validates state == Disputed)
- `resolver_registry: Account<ResolverRegistry>`
- Remaining accounts: 1 Resolver PDA (the new juror) for stake snapshot + active_disputes increment

Errors to add:
- `TiebreakerAlreadyAdded` ("Tiebreaker juror has already been added")
- `DisputeNotTied` ("Dispute is not in a tie state")
- `QuorumNotReached` ("Not enough votes to finalize dispute")

**2. settle_dispute_rewards.rs:**

Called after dispute is settled (dispute.status == 2, market.state == Resolved). This instruction processes slashing and rewards for jurors. It should be called once per juror.

The instruction takes `juror_index: u8` to identify which juror to settle.

Steps:
1. Validate `dispute.status == 2` (Settled) and `market.state == 2` (Resolved).
2. Determine if the juror at `juror_index` is majority or minority:
   - If juror voted (bit set in `votes_submitted`): check which side won. Since individual votes are private, we use a simplification for v1: **all voters on the losing side are minority**. BUT we cannot know individual votes on-chain after MPC.

   **V1 simplification:** Instead of per-juror slashing (which requires knowing individual votes), implement a simpler model:
   - Non-voters (bit NOT set in `votes_submitted`): slash 0.5% from their Resolver stake.
   - All voters: no individual slash in v1 (we cannot determine majority/minority without revealing individual votes).
   - Slashed amounts from non-voters are distributed equally to all voters.

   This preserves vote privacy while still penalizing non-participation. Per-CONTEXT.md's "Claude's Discretion" section, the exact incentive mechanics are at Claude's discretion.

3. Transfer slashed USDC from non-voter resolver vaults to voter resolver vaults (or voter wallet ATAs).
4. Decrement `active_disputes` on each processed resolver.

Accounts:
- `payer: Signer` (mut)
- `dispute: Account<Dispute>` (read-only, for juror list and votes_submitted)
- `market: Account<Market>` (validates Resolved state)
- `resolver: Account<Resolver>` (mut, the juror being settled)
- `resolver_vault: Account<TokenAccount>` (mut, resolver's staked USDC)
- If non-voter: recipient token accounts for distributing slash
- `token_program`
