**2. Create useOpenDispute hook (`app/src/hooks/useOpenDispute.ts`):**

Follow the `usePlaceBet` pattern (TanStack Query `useMutation`, step-based progress, exponential backoff). This hook does TWO chained transactions:

**Step tracking type:**
```typescript
export type EscalateStep =
  | 'idle'
  | 'checking'     // Pre-checking resolver count
  | 'escalating'   // Step 1/2: open_dispute TX
  | 'initializing' // Step 2/2: init_dispute_tally TX
  | 'confirming'
  | 'retrying'
  | 'success'
  | 'error'
```

**Hook signature:** `useOpenDispute(marketId: number)`

**Mutation flow:**

Step 1 -- Pre-check resolver count:
- Set step to `'checking'`
- Use `useReadOnlyProgram` or inline fetch to read `ResolverRegistry` PDA via `program.account.resolverRegistry.fetch(getResolverRegistryPda()[0])`
- If `resolvers.length < 7`, throw error "Not enough resolvers available ({count}/7 required)"

Step 2 -- Submit open_dispute TX (`'escalating'` step):
- Derive PDAs: marketPda, positionPda (user's), resolverRegistryPda, disputePda, disputeTallyPda
- Fetch ResolverRegistry account to get the list of active resolver pubkeys
- For each resolver pubkey, derive their Resolver PDA via `getResolverPda(resolverPubkey)` -- these become `remaining_accounts`
- Build remaining accounts array: `resolvers.map(pubkey => ({ pubkey: getResolverPda(pubkey)[0], isSigner: false, isWritable: true }))`
- Submit: `program.methods.openDispute().accounts({ escalator: publicKey, market: marketPda, userPosition: positionPda, resolverRegistry: resolverRegistryPda, dispute: disputePda, disputeTally: disputeTallyPda, systemProgram }).remainingAccounts(resolverAccounts).rpc()`

Step 3 -- Auto-chain init_dispute_tally (`'initializing'` step):
- Generate random `computationOffset` via `crypto.getRandomValues`
- Dynamic import `@arcium-hq/client` for Arcium account derivation (same pattern as useCastVote/useComputePayouts)
- Derive Arcium accounts with comp_def for `'init_dispute_tally'` circuit
- Submit: `program.methods.initDisputeTally(computationOffset).accounts({ payer: publicKey, dispute: disputePda, disputeTally: disputeTallyPda, systemProgram, arciumProgram, ...arciumAccounts }).rpc()`

Step 4 -- Confirm and invalidate:
- Invalidate `['dispute', marketId]` and `['market', marketId]` queries
- Toast success: "Dispute escalated -- jury voting will begin shortly"

**Error handling:**
- If open_dispute succeeds but init_dispute_tally fails: show error toast with retry button ("Dispute opened -- initializing vote tally failed. Retry?"). The retry should ONLY retry init_dispute_tally since the dispute is already open on-chain.
- Same exponential backoff for MPC lock contention as usePlaceBet (5 attempts: 2s, 4s, 8s, 16s, 32s) with toast "Dispute busy -- retrying... (attempt X)"
- On init_dispute_tally retry: only re-submit init_dispute_tally, not open_dispute (dispute already exists)

**Return:** `{ ...mutation, step, retryCount, reset, resolverCount }` where `resolverCount` is fetched on mount for the pre-check.
