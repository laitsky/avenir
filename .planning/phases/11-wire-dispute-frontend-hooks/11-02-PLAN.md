**1. Create useFinalizeDispute hook (`app/src/hooks/useFinalizeDispute.ts`):**

Follow the `useComputePayouts` pattern closely -- finalize_dispute is also a permissionless MPC queue instruction that any wallet can trigger once quorum is reached.

**Hook signature:** `useFinalizeDispute(marketId: number)`

Uses TanStack Query `useMutation` (same as useComputePayouts).

**Mutation flow:**
1. Derive PDAs: marketPda, disputePda, disputeTallyPda
2. Generate random `computationOffset` via `crypto.getRandomValues(new Uint8Array(8))`
3. Dynamic import `@arcium-hq/client` for Arcium account derivation
4. Get Arcium env, cluster offset, comp_def index for `'finalize_dispute'` circuit
5. Build arciumAccounts object (mxeAccount, signPdaAccount, mempoolAccount, executingPool, computationAccount, compDefAccount, clusterAccount, poolAccount, clockAccount, arciumProgram)
6. Submit: `program.methods.finalizeDispute(computationOffset).accounts({ payer: publicKey, dispute: disputePda, disputeTally: disputeTallyPda, market: marketPda, systemProgram, arciumProgram, ...arciumAccounts }).rpc({ commitment: 'confirmed' })`
7. Return sig

**Callbacks:**
- `onSuccess`: Invalidate `['dispute', marketId]` and `['market', marketId]` queries. Toast: "Revealing outcome -- fog clearing!"
- `onError`: Toast: "Failed to reveal outcome: {msg}". Include MPC lock contention detection (same as usePlaceBet): if error includes 'MpcLocked' or 'mpc_lock', show "Dispute busy -- try again in a moment" with retry action in toast.

**Return:** Standard `useMutation` return (isPending, mutate, etc.)

**2. Create useAddTiebreaker hook (`app/src/hooks/useAddTiebreaker.ts`):**

Simpler hook -- add_tiebreaker is NOT an MPC instruction. It is a plain Anchor instruction (no Arcium accounts needed).

**Hook signature:** `useAddTiebreaker(marketId: number)`

Uses TanStack Query `useMutation`.

**Mutation flow:**
1. Derive PDAs: disputePda, marketPda, resolverRegistryPda
2. Fetch ResolverRegistry to get resolver list (same pattern as useOpenDispute)
3. Fetch Dispute account to get current jurors list
4. Find the first resolver in registry NOT already in dispute.jurors -- this is the tiebreaker candidate
5. Derive that candidate's Resolver PDA via `getResolverPda(candidatePubkey)`
6. Submit: `program.methods.addTiebreaker().accounts({ payer: publicKey, dispute: disputePda, market: marketPda, resolverRegistry: resolverRegistryPda }).remainingAccounts([{ pubkey: candidateResolverPda, isSigner: false, isWritable: true }]).rpc({ commitment: 'confirmed' })`

Note: The on-chain instruction does its own deterministic juror selection -- the remaining_accounts just needs to include the correct Resolver PDA. However, the client cannot predict which juror the on-chain code will select (it uses clock.slot for the LCG seed). So the client should pass ALL non-juror resolver PDAs as remaining_accounts, and the on-chain code will pick from them.

Actually, re-reading add_tiebreaker.rs more carefully: it only expects 1 remaining_account (the specific resolver PDA for the selected juror). But the selection is deterministic using `clock.slot` which the client cannot predict. Since the on-chain code validates the PDA matches, we need a different approach:

**Simpler approach:** Pass all candidate (non-juror) resolver PDAs as remaining_accounts. The on-chain code only reads `remaining_accounts.first()` -- so we need to pass exactly the right one. Since we cannot predict `clock.slot`, we should pass the resolver PDAs in the same order as the registry so the on-chain LCG can find its target.

**Pragmatic solution:** The on-chain `add_tiebreaker` selects a juror index deterministically, then reads `remaining_accounts.first()` and validates it matches. The client needs to simulate the same selection. Since `clock.slot` changes, this is inherently racy. The safest approach: pass ALL non-juror resolver PDAs as remaining accounts and handle the error if the slot-based selection doesn't match. On retry (which the user can do manually), the slot will be different and may succeed. Include a note about this in the error toast.

**Actually**, looking at the code again more carefully: the on-chain code picks `selected_index` from registry, gets `new_juror_pubkey = registry.resolvers[selected_idx]`, then validates `remaining_accounts.first().key() == expected_pda`. So the client MUST pass the correct one. Since slot is unpredictable, the practical approach is:

1. Fetch current dispute jurors
2. Fetch registry resolvers
3. Filter out existing jurors to get candidates
4. If only 1 candidate, pass that one (guaranteed match)
5. If multiple candidates, pass the first non-juror resolver PDA and hope slot alignment works. If TX fails, retry with toast.

This is an edge case for v1 -- in practice, the tiebreaker selection will work on retry since slot changes.

**Callbacks:**
- `onSuccess`: Invalidate `['dispute', marketId]` queries. Toast: "Tiebreaker juror added -- voting extended 24h"
- `onError`: Toast: "Failed to add tiebreaker: {msg}" with retry action

**Return:** Standard `useMutation` return.

**3. Add tie detection and auto-tiebreaker:**

Per CONTEXT.md: "Tie scenario: wire useAddTiebreaker hook -- when dispute resets to Voting after tie, show 'Tie detected -- selecting tiebreaker juror' and auto-trigger add_tiebreaker"

Tie detection: When `dispute.status === 0` (Voting) AND `dispute.voteCount >= dispute.quorum` AND the dispute was previously in Finalizing, the finalize_dispute_callback has detected a tie and reset to Voting. The observable signal is:
- `dispute.status === 0` (reset to Voting)
- `dispute.voteCount > 0` (has votes already)
- `dispute.tiebreakerAdded === false` (hasn't added tiebreaker yet)
- `dispute.voteCount >= dispute.quorum` (voting had completed)

This state means a tie was detected. In `DisputePendingMode`, detect this condition and auto-trigger `useAddTiebreaker`:

```typescript
const isTieDetected = dispute.status === 0
  && dispute.voteCount >= dispute.quorum
  && !dispute.tiebreakerAdded
```

- Show: "Tie detected -- selecting tiebreaker juror"

- During mutation: show spinner + "Adding tiebreaker juror..."
- On success: dispute will update via websocket (jurors array grows, quorum increments, voting_end extended)
