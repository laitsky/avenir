Create `tests/place-bet.ts` with integration tests for the `place_bet` instruction. Use the same test infrastructure as `tests/avenir.ts` (AnchorProvider, workspace, etc.) and reuse MPC helpers from `tests/mpc/helpers.ts` where applicable.

**Test setup (before hook):**
- Create USDC mint (6 decimals)
- Create a bettor keypair, airdrop SOL
- Create a creator keypair, airdrop SOL
- Initialize Config PDA (or reuse if the test suite shares state)
- Whitelist the creator
- Create a market with future deadline (e.g., current time + 3600)
- Initialize MarketPool via init_pool MPC (or set up the market_pool to have valid encrypted zeros)
- Create bettor's USDC associated token account and mint test USDC (e.g., 100 USDC = 100_000_000 token units)
- Set up Arcium context for MPC account derivation

**Helper functions:**
- `getPositionPda(marketId, user)` -- derives UserPosition PDA
- `getMarketPoolPda(marketId)` -- derives MarketPool PDA
- Reuse `getMarketPda`, `getVaultPda` patterns from avenir.ts

**Test cases:**

1. **"places a valid Yes bet with USDC transfer"** (happy path)
   - Call `program.methods.placeBet(new BN(5_000_000), true, computationOffset, isYesCiphertext, amountCiphertext, pubKey, nonce)` with all required accounts
   - Verify: bettor's token balance decreased by 5 USDC
   - Verify: vault balance increased by 5 USDC
   - Verify: market.mpcLock is true
   - Verify: market.pendingBettor equals bettor's pubkey
   - Verify: market.pendingAmount equals 5_000_000
   - Verify: market.pendingIsYes is true
   - Verify: market.lockTimestamp is set (> 0)
   - Verify: UserPosition PDA exists (created by init_if_needed)

2. **"rejects bet below minimum (1 USDC)"**
   - Call placeBet with amount = 999_999 (below 1 USDC minimum)
   - Expect error containing "BetTooSmall" or error code matching the BetTooSmall variant
   - Verify: no USDC transferred, market state unchanged

3. **"rejects bet on expired market"**
   - Create a market with a past deadline (resolution_time in the past)
   - Call placeBet on this expired market
   - Expect error containing "MarketExpired"
   - Note: This requires creating a second market with a past deadline. Since the market creation instruction validates deadline > now + 1h, you may need to create a market with a near-future deadline and then manipulate Clock or simply test that the validation logic exists by inspecting the error. Alternative: skip this test if Clock manipulation is too complex on localnet, and add a comment explaining why.

4. **"rejects bet when MPC lock is active"**
   - After test 1 (which sets mpc_lock), attempt another placeBet on the same market
   - Expect error containing "MpcLocked"
   - Verify: second bettor's USDC balance unchanged

5. **"rejects bet on opposite side of existing position"**
   - This requires a completed first bet (callback must have run). If MPC callback cannot be triggered in tests without Arcium running:
     - Option A: If init_pool + MPC callback infrastructure is available from Phase 3 tests, use it to complete a full bet cycle first, then test opposite-side rejection
     - Option B: Manually set UserPosition fields via a test-only mechanism or create the test with a comment noting it requires MPC infrastructure
   - If implementing: bettor has a Yes position -> attempt No bet -> expect WrongSide error

6. **"validates UserPosition PDA creation via init_if_needed"**
   - After test 1, fetch the UserPosition account at the derived PDA
   - Verify: account exists, market_id is set, user is default (will be set by callback), yes_amount and no_amount are 0 (callback hasn't run yet)

**NOTE on MPC integration:** Some tests (like verifying callback behavior) require the Arcium MPC cluster to be running. For tests that only verify `place_bet` behavior (not callback), the test can verify state after `place_bet` returns but before the MPC callback runs. The callback tests were already covered in Phase 3's test infrastructure.
