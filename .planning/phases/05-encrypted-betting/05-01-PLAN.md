This is NOT inside the `mpc/` subdirectory -- it lives at the top-level `instructions/` directory because it's a user-facing instruction (not an MPC-internal one).

The handler function signature:
```rust
pub fn handler(
    ctx: Context<PlaceBet>,
    amount: u64,
    is_yes: bool,
    computation_offset: u64,
    is_yes_ciphertext: [u8; 32],
    amount_ciphertext: [u8; 32],
    pub_key: [u8; 32],
    nonce: u128,
) -> Result<()>
```

**Handler logic (in this exact order):**

1. **Get clock:** `let clock = Clock::get()?;`

2. **Validate bet amount:** `require!(amount >= 1_000_000, AvenirError::BetTooSmall);`

3. **Validate market state:** `require!(market.state == 0, AvenirError::MarketNotOpen);`

4. **Validate deadline:** `require!(clock.unix_timestamp < market.resolution_time, AvenirError::MarketExpired);`

5. **Check lock with timeout recovery:**
   - If `market.mpc_lock` is true:
     - If `clock.unix_timestamp - market.lock_timestamp > 60`: lock is stale. Refund the stuck bettor's USDC from vault using PDA-signed token::transfer (vault -> pending_bettor_token_account). Then clear: `mpc_lock = false`, `pending_bettor = Pubkey::default()`, `pending_amount = 0`, `pending_is_yes = false`, `lock_timestamp = 0`. The `pending_bettor_token_account` is a separate account in the struct for this purpose.
     - Else: return `Err(AvenirError::MpcLocked.into())` -- lock is active, reject immediately.
   - The PDA signer seeds for vault refund: `&[b"market", &market_id.to_le_bytes(), &[market.bump]]` (Market PDA is vault authority).

6. **Validate side consistency:** Check if UserPosition already has a non-zero position. If `user_position.yes_amount > 0` and `!is_yes`, reject with WrongSide. If `user_position.no_amount > 0` and `is_yes`, reject with WrongSide. (If position is fresh/zero, any side is allowed.)

7. **Transfer USDC: user -> vault:**
   ```rust
   let cpi_accounts = Transfer {
       from: ctx.accounts.user_token_account.to_account_info(),
       to: ctx.accounts.market_vault.to_account_info(),
       authority: ctx.accounts.bettor.to_account_info(),
   };
   token::transfer(CpiContext::new(ctx.accounts.token_program.to_account_info(), cpi_accounts), amount)?;
   ```

8. **Store pending bet data on Market:**
   ```rust
   market.pending_bettor = ctx.accounts.bettor.key();
   market.pending_amount = amount;
   market.pending_is_yes = is_yes;
   market.mpc_lock = true;
   market.lock_timestamp = clock.unix_timestamp;
   ```

9. **Build ArgBuilder and queue MPC computation:**
   Use the same ArgBuilder pattern as the existing `update_pool.rs`:
   ```rust
   let args = ArgBuilder::new()
       .x25519_pubkey(pub_key)
       .plaintext_u128(nonce)
       .encrypted_bool(is_yes_ciphertext)
       .encrypted_u64(amount_ciphertext)
       .account(ctx.accounts.market_pool.key(), 16, 64)
       .build();
   ```

10. **Build extended CallbackAccount vector:**
    The callback needs access to: MarketPool (writable), Market (writable), UserPosition (writable), market_vault (writable), user_token_account (writable -- for failure refund), token_program (not writable).
    ```rust
    let callback_accounts = vec![
        CallbackAccount { pubkey: ctx.accounts.market_pool.key(), is_writable: true },
        CallbackAccount { pubkey: ctx.accounts.market.key(), is_writable: true },
        CallbackAccount { pubkey: ctx.accounts.user_position.key(), is_writable: true },
        CallbackAccount { pubkey: ctx.accounts.market_vault.key(), is_writable: true },
        CallbackAccount { pubkey: ctx.accounts.user_token_account.key(), is_writable: true },
        CallbackAccount { pubkey: ctx.accounts.token_program.key(), is_writable: false },
    ];
    ```

11. **Build callback instruction and queue computation:**
    ```rust
    let callback_ixs = vec![UpdatePoolCallback::callback_ix(
        computation_offset, &ctx.accounts.mxe_account, &callback_accounts,
    )?];
    queue_computation(ctx.accounts, computation_offset, args, callback_ixs, 1, 0)?;
    ```

**PlaceBet accounts struct:**
Use `#[queue_computation_accounts("update_pool", bettor)]` macro (same circuit as update_pool, bettor is payer).

Required accounts:
- `bettor: Signer<'info>` (mut, payer for computation + UserPosition rent)
- `market: Account<'info, Market>` (mut, seeds = [b"market", market.id.to_le_bytes()], bump = market.bump)
- `market_pool: Account<'info, MarketPool>` (seeds = [b"market_pool", market_pool.market_id.to_le_bytes()], bump = market_pool.bump, constraint = market_pool.market_id == market.id)
- `user_position: Account<'info, UserPosition>` (init_if_needed, payer = bettor, space = 8 + UserPosition::INIT_SPACE, seeds = [b"position", market.id.to_le_bytes(), bettor.key()], bump)
- `user_token_account: Account<'info, TokenAccount>` (mut, token::mint = usdc_mint, token::authority = bettor) -- bettor's USDC account
- `market_vault: Account<'info, TokenAccount>` (mut, seeds = [b"vault", market.id.to_le_bytes()], bump = market.vault_bump, token::mint = usdc_mint, token::authority = market)
- `pending_bettor_token_account: Account<'info, TokenAccount>` (mut) -- token account for timeout refund target. Validated against market.pending_bettor only when timeout recovery triggers. Use `/// CHECK:` if needed, or use UncheckedAccount and validate manually in handler when timeout path is taken.

  **IMPORTANT:** The `pending_bettor_token_account` is tricky. When there is no stale lock, this account is unused but still must be passed. Use `UncheckedAccount` with `/// CHECK: Only used for timeout refund -- validated against market.pending_bettor when timeout recovery triggers`. This avoids requiring a valid TokenAccount when no timeout exists.

  Actually, a simpler approach: use `/// CHECK:` UncheckedAccount for `pending_bettor_token_account`. In the timeout refund path, deserialize it manually as a token account and validate ownership. When no timeout, it's ignored.

- `token_program: Program<'info, Token>`
- `system_program: Program<'info, System>` -- for init_if_needed
