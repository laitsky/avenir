Output: Fixed encryption.ts with deterministic nonce derivation (nonce + 1 for second field).

**CRITICAL NOTE from research (Open Question 1):**
The ArgBuilder on-chain currently passes a single `.plaintext_u128(nonce)` for the entire `Enc<Shared, BetInput>` struct. When we change encryption.ts to use two nonces, the on-chain ArgBuilder in `place_bet.rs` may also need updating to pass `.plaintext_u128(nonce1).plaintext_u128(nonce2)`. However, per research recommendation: try the nonce+1 approach first with the existing single nonce on-chain. The ArgBuilder may auto-derive sequential nonces internally. If MPC decryption fails in testing, the on-chain change is documented but deferred (since DKG blocker prevents actual MPC testing anyway).

The return type MUST include both nonces so that if on-chain changes ARE needed later, the consumer (usePlaceBet) has both values available.

**Step 1: Fix encryptBetForMPC nonce derivation**

Change lines 48-54 of encryption.ts. The current code generates one nonce and uses it for both cipher.encrypt calls. Fix:

1. Generate the first nonce normally:
   ```typescript
   const nonce1 = new Uint8Array(16)
   crypto.getRandomValues(nonce1)
   ```

2. Derive second nonce deterministically as nonce1 + 1 (little-endian u128 increment):
   ```typescript
   const nonce2 = new Uint8Array(16)
   nonce2.set(nonce1)
   let carry = 1
   for (let i = 0; i < 16 && carry; i++) {
     const sum = nonce2[i] + carry
     nonce2[i] = sum & 0xff
     carry = sum >> 8
   }
   ```

3. Use separate nonces for each field:
   ```typescript
   const isYesCiphertext = cipher.encrypt([BigInt(isYes ? 1 : 0)], nonce1)
   const amountCiphertext = cipher.encrypt([BigInt(amount)], nonce2)
   ```

4. Update the return type to include both nonces:
   ```typescript
   return {
     isYesCiphertext: isYesCiphertext[0],
     amountCiphertext: amountCiphertext[0],
     publicKey,
     nonce: nonce1,        // Primary nonce (backward compat)
     nonceBN: deserializeLE(nonce1),   // Primary nonce as BigInt
     nonce2,               // Second nonce for amount field
     nonce2BN: deserializeLE(nonce2),  // Second nonce as BigInt
   }
   ```

5. Update the return type in the function signature Promise type to include `nonce2: Uint8Array` and `nonce2BN: bigint`.

6. Update the JSDoc comment at the top to document the nonce fix:
   - Remove the note about encrypting fields "separately" with no mention of nonce safety
   - Add: "Each field is encrypted with a unique nonce: nonce1 for isYes, nonce1+1 for amount. This prevents XOR-based plaintext relationship leakage with stream ciphers."
