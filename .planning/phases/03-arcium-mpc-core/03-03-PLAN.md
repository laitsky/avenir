<task type="auto">
  <name>Task 1: Create reusable MPC test helpers</name>
  <files>tests/mpc/helpers.ts</files>
  <action>
Create `tests/mpc/helpers.ts` with reusable utilities that Phases 5, 6, 8 can import:

**setupArciumContext(provider, programId)**
- Gets ArciumEnv (cluster offset, etc.)
- Gets MXE public key
- Returns { arciumEnv, mxePublicKey } for subsequent operations

**encryptBetInput(mxePublicKey, isYes: boolean, amount: number)**
- Generates x25519 keypair
- Derives shared secret with MXE
- Creates RescueCipher
- Encrypts isYes (as BigInt 0 or 1) and amount (as BigInt)
- Generates random nonce
- Returns { ciphertext: [Uint8Array, Uint8Array], publicKey, nonce, computationOffset }
  (computationOffset = random BN for computation account)

**awaitAndVerifyCallback(provider, computationOffset, programId)**
- Calls `awaitComputationFinalization()`
- Returns finalization result (or throws on timeout)

**createTestMarket(program, admin, config, usdcMint, ...)**
- Creates a market with standard test params (question, category, deadline, resolution_source)
- Returns { marketPda, marketPoolPda, marketId }
- Handles createMint setup if needed
- This encapsulates the setup boilerplate so MPC tests focus on the MPC part

**getArciumAccounts(program, arciumEnv, computationOffset, compDefName)**
- Returns the standard Arcium account set: computationAccount, clusterAccount, mxeAccount, mempoolAccount, executingPool, compDefAccount
- These are needed for every queue_computation call

**initCompDef(program, payer, compDefName)**
- Generic comp_def initialization helper
- Calls the appropriate `program.methods.init{Name}CompDef()` based on compDefName

All helpers should use `@arcium-hq/client` imports exclusively (no wrapper layers, per CONTEXT.md).
Import types from `@coral-xyz/anchor` and `@solana/web3.js`.

**Test 1: "init_pool creates zero-valued encrypted pool state"**
1. Create a market via create_market (which creates Market + MarketPool PDAs)
2. Queue init_pool computation for this market
3. Await callback finalization
4. Fetch MarketPool account
5. Verify: yes_pool_encrypted and no_pool_encrypted are non-zero (they are encrypted zeros, not literal zero bytes)
6. Verify: nonce was written (non-zero)

**Test 2: "update_pool accumulates a single bet correctly"**
1. Use the market from Test 1 (pool already initialized)
2. Encrypt a bet input: { is_yes: true, amount: 1_000_000 } (1 USDC)
3. Queue update_pool computation with encrypted ciphertext
4. Await callback finalization
5. Fetch MarketPool account
6. Verify: yes_pool_encrypted changed from initial value (ciphertext updated)
7. Verify: nonce was updated (different from init_pool nonce)
8. Fetch Market account
9. Verify: sentiment is 1 (Leaning Yes -- only bet is on Yes side)
10. Verify: mpc_lock is false (released by callback)
11. Verify: total_bets is 1

**Test 3: "update_pool handles sequential bets and sentiment transitions"**
1. Use the same market (pool has 1 USDC on Yes)
2. Encrypt a second bet: { is_yes: false, amount: 2_000_000 } (2 USDC on No)
3. Queue and await update_pool
4. Verify: sentiment is 3 (Leaning No -- 2 USDC No vs 1 USDC Yes)
5. Verify: total_bets is 2
6. Verify: nonce was incremented again
7. Encrypt a third bet: { is_yes: true, amount: 1_000_000 } (1 USDC on Yes)
8. Queue and await update_pool
9. Verify: sentiment is 2 (Even -- 2 USDC on each side)
10. Verify: total_bets is 3

**Test 4: "update_pool rejects when mpc_lock is active"**
1. This tests the sequential lock
2. Set mpc_lock = true manually on the Market account (or queue a computation and immediately try another before callback)
3. Attempt to queue update_pool
4. Verify: transaction fails with MpcLocked error

**Key assertions for the POC:**
- Ciphertext bytes on MarketPool change after each MPC call (proving MPC computed something)
- Nonce increments with each call (proving MXE processed the ciphertext)
- Sentiment transitions correctly through the 3 states (proving circuit logic works)
- mpc_lock prevents concurrent access (proving sequential lock works)

These tests validate INF-03 (encrypted state relay pattern -- ciphertext stored on-chain, passed to MPC, updated via callback).
